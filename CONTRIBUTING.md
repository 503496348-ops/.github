# Contributing Guide

## 基本原则

- 一个 PR 只解决一个明确问题。
- 变更必须可复现、可审计、可回滚。
- 不提交密钥、token、密码、私钥、客户数据或内部服务器信息。
- 竞品融合必须保留许可证合规信息，但不得在产品文案中残留“融合自”、外部作者名、GitHub URL、竞品变量名等污染痕迹。

## 推荐流程

1. 先创建或关联 Issue，说明目标、范围和验收标准。
2. 新建短生命周期分支：`feat/<topic>`、`fix/<topic>`、`docs/<topic>`。
3. 小步提交，提交信息使用：`feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`chore:`。
4. 提交 PR 前完成自检：
   - 是否能运行/构建/测试。
   - 是否更新 README 或必要文档。
   - 是否无密钥、无外部污染引用。
   - 是否符合仓库许可证要求。
5. PR 说明必须包含：变更摘要、验证证据、风险与回滚方式。

## 验收底线

- 没有验证证据的 PR 不合并。
- 安全、密钥、许可证、外部引用污染属于 P0/P1 问题，必须先修。
- 大规模重命名、删除、归档、历史改写必须单独审批。
